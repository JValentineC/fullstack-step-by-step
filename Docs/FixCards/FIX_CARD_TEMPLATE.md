Overview and Steps:
Skill Support Loop
A 15-minute workflow to unblock interns and confirm proficiency   
Expectations for Tech Team Use
Everyone uses the Fix Card format
Everyone ends with the Prove step.
One group = one failure mode.
One small group = one fix type.
Release based on Prove evidence, not time or vibes
Logistics
One skill/task to focus on
A way to verify success (before & after check, expected output, checklist, or behavior check)
15 minutes to meet with the small group
3–6 interns with the same failure mode
The Workflow
Select your focus skill or task
Define “Done” (30s)
Write one sentence that includes the action + the proof.
Done = They can ___ AND it passes ___.  (Action + proof)
Examples:
Done = They can filter the dataset correctly and the row count matches expected.
Done = They can set the permission correctly and access is denied/allowed exactly as intended.
Done = They can build the chart correctly and it meets the 3-item quality check
Rule: If you can’t define “Done,” don’t run a group yet.
Sort People into Buckets Quickly (2m)
Use artifacts or a quick attempt to sort into one of three buckets:
Broken: does not meet Done
Almost: meets part of Done, fails 1 thing
Solid: meets Done
Rule: Run a group only if the bucket shares the same failure mode. If they’re broken for different reasons, split the group.
Pick ONE Fix Type (30s)
Choose one fix type for your small group planning.
Explain = they don’t understand the rule/why
Walkthrough = they missed steps/order
Practice = they need reps/consistency/speed
Troubleshoot = they can’t isolate the cause
Upgrade = they’re solid; push quality/transfer
Rule: Don’t mix; plan separate groups for separate fix types.
Run a 15-minute block using the Show → Do → Prove structure
0–3 min: SHOW: Demonstrate one correct example OR one common wrong example + fix.
“Here’s the target behavior.”
“Here’s the typical mistake.”
“Here’s the one move that fixes it.”
3–11 min: DO: Interns do the same thing on a new item while you coach.
Pair them if helpful: Driver executes, Navigator reads criteria and calls out checks.
Keep them moving. Avoid long explanations.
11–15 min: PROVE: Run the verification check.
Output / Checklist / Behavior Check / Before & After Check
Expected Output: You define a specific, correct result in advance. The intern’s work must match it.
Best for: Anything with a clear “right answer” (data tasks, calculations, queries, reports, configurations that produce a known result).
How to Run It;
Give an input (or dataset).
Specify the expected result (number, row count, list of items, screenshot reference).
Intern runs their process and compares.
3 Item Checklist: A short list of visible criteria tied to the skill. Intern passes if all three are met.
Best for: Skills where correctness includes quality/standards, not just output (documentation, dashboards, UX reviews, security hygiene, code readability, project structure).
How to Run It:
Choose 3 criteria that matter most.
Review the artifact against them.
Mark pass/fail for each.
Behavior Check: Instead of a numeric output, you verify the system behaves correctly under a specific scenario.
Best for: Permissions, deployments, integrations, UI flows, system settings, tool administration, cloud resources—anything where “correct” is how it behaves, not a number.
How to Run It:
Define the scenario: “When we do X, the system should do Y.”
Intern demonstrates it live (or via a recorded check).
Confirm pass/fail.
Before & After Check: You compare the artifact before the small-group block to the revised version after to confirm measurable improvement on specific dimensions.
Best for: Quality upgrades—readability, maintainability, performance, clarity, accessibility, security, organization.
How to Run It:
Pick 1–2 improvement targets (e.g., “clarity” and “robustness”).
Have the intern make changes.
Show a simple diff: what changed and why it’s better.
Rule: Every group ends with proof. If you didn’t verify, you don’t know if it worked.
Decide next step (30s)
Pass Done check → release to main work OR give an Upgrade task
Fail Done check → choose one:
Loop once with a tighter scaffold (smaller task, clearer steps), OR
Assign a targeted redo and schedule a fast recheck
Rule: Don’t keep them in the group without changing the tactic.

Minimal Scripts → Support Tool for What to Say
Start: “Goal is Done: ___ . We’re going to Show → Do → Prove. We finish when the check passes.”
If they want step-by-step: “I’ll show one example. Then you do it. I’ll coach, but you drive.”
If stuck: “What’s your hypothesis? What’s the smallest test to confirm it?”
Close: “Run the proof. If it passes, you’re released. If not, we loop once with a tighter scaffold.”
Quick Checklists Templates

Troubleshooting loop (for any tool)
Hypothesis stated (before changes)
Smallest test run (change one variable)
Result logged → next step chosen
